The Problem

Congestion dominates
everything else

The placement proxy is a weighted sum of wirelength, density, and congestion. After normalization, congestion is ~30× larger than wirelength. Classic EDA tools optimize wirelength — which actively hurts this proxy by clustering macros into congested corridors.

  • Wirelength-only SA clusters macros → congestion spikes → proxy regresses
  • initial.plc already has hand-tuned spread — preserve it, don't rebuild from scratch
  • Every directed move in CongFlow targets the real H/V routing congestion map
  • Density fallback rankers are anti-correlated with proxy (empirically verified)

Algorithm

Nine-phase congestion engine

Directed phases (1–5) use live H/V routing congestion maps from TILOS PlacementCost. Exploratory phases fill the remaining time budget. All candidates are re-legalized and scored exactly — lowest proxy wins.

0
Baseline Legalization
ALWAYS RUNS
Vectorized spiral-search legalization from initial.plc, placing macros largest-area-first. Critical float32 precision fix preserves bit-equivalence with the scalar baseline trajectory — without it, ibm04 diverges to 1.3364 instead of 1.3316. 12× faster (3.2s → 0.27s on ibm04).
1
Iterative Congestion-Gradient Descent
PHASE 1
Up to 12 steps at frac=0.04 with adaptive halving. After each improving step, both the source position and the live plc congestion map update for the next iteration. Stops when a step fails to improve or budget < 3×t_score.
↓ ibm04: 1.4101 → 1.3316 via 7 iterations
2
Wide Steps from Baseline
PHASE 2
Larger steps (frac=0.08, 0.12) from baseline_pos using the stale congestion map from Phase 1's best iteration. The stale map acts as second-order curvature information, guiding larger jumps into unexplored basins.
↓ ibm02: 1.6800 → 1.5823
3
Best-Position Cong-Grad
PHASE 3
Perturbs the current best_pl at frac=0.04 using a cached congestion map. Finds basins missed by Phases 1/2. ibm04's stable 1.3316 win lives specifically in this basin — it requires starting from the Phase 1 best rather than the original baseline.
5
Async DREAMPlace — Best-of-Both
PHASE 5 · v2 ENHANCED
Two DREAMPlace instances launched at place() entry, running in parallel with Phases 1–3. "fixed" uses soft_macros_movable=False; "movable" lets DP's NLP jointly optimize soft macros. Where initial.plc density < 0.85, movable DP spreads softs for massive wins. Best-of-both selection recovers regressions on dense benchmarks.
↓ −0.0161 avg · ibm06: −0.121, ibm03: −0.103, ibm02: −0.051
7
DP-Rescue Cong-Grad Chain
PHASE 7
After the noise loop, for each completed DP placement, runs a greedy cong-grad chain (up to 3 iters). DREAMPlace's plc-state mutation seeds new cong-grad basins not reachable from initial.plc. 19% win rate: ibm10 −0.045, ibm02 −0.035, ibm04 −0.018.
8
TOP-K Cong-Grad + Proxy 2-opt
PHASE 8 · NEW IN v2
(a) TOP-K cong-grad: target the K highest-congestion macros from best_pl, apply directed perturb. (b) Proxy-driven 2-opt (k=10 neighbors, 6 iters, 15s budget): each swap is accepted or rejected based on actual proxy cost — not displacement. v1's displacement criterion was anti-correlated with proxy, wasting the entire 15s budget.
↓ A1: −0.0059 avg · All 17 benchmarks improve
9
Random-Order Legalization
PHASE 9 · NEW IN v2
Shuffles the macro placement order in the spiral-search legalizer, exploring alternative macro assignment topologies. The greedy spiral-search is order-sensitive — different orderings produce different legal arrangements from the same initial positions.
↓ −0.0003 avg

v2 Key Changes

What made the difference

Five targeted changes moved the average from 1.4804 → 1.4486, a −0.0318 improvement across 17 benchmarks.

🎯
Proxy-Driven 2-opt
Issue A1 · _two_opt_proxy_swap

v1's 2-opt accepted swaps when per-pair displacement decreased — a criterion with no relation to proxy cost. On ibm01/04/10 it actively made proxy worse while appearing to "improve" the placement. v2 tentatively applies each swap, scores it with the real proxy, and accepts only if cost strictly decreases.

↓ −0.0059 avg · All 17 benchmarks improve
🔀
Best-of-Both DREAMPlace
Issue A2 · soft_macros_movable=True

Two DP configs run in parallel: "fixed" (soft macros pinned at initial positions) and "movable" (DP NLP jointly optimizes soft macros). Where canvas density < 0.85, movable DP spreads stdcell clusters to dramatically lower congestion. Best-of-both selection prevents regressions on already-dense benchmarks (ibm01, ibm09).

↓ −0.0161 avg · ibm06 −0.121, ibm03 −0.103, ibm02 −0.051
⚡
4-Phase Incremental Scorer
Issue B3 (phases 1–4) · IncrementalScorer

Four successive speedups: (1) global position cache eliminates Python get_pos loops; (2) per-net HPWL with macro→nets index updates only touched nets; (3) numpy ABU skips .tolist() overhead; (4) per-net incremental routing footprint cache. More 2-opt iterations per budget window directly translates to better scores.

22ms → 5ms per score · --all wall-clock −155s
🌡️
TOP-K Congestion-Gradient Phase
Issue A6 · Phase 8 TOP-K cong-grad

After Phase 7, identifies the K macros with the highest per-cell congestion contribution and applies a targeted cong-grad perturbation from best_pl. Focuses the perturbation budget on the macros actually causing the congestion bottleneck rather than scattering gradient across the full canvas.

↓ −0.0010 avg · Wins on ibm02/03/04/06/16
⏱️
Cumulative Wall-Clock Guard
Issue B1 · cumulative_elapsed tracking

v1's per-benchmark budget didn't account for cumulative --all runtime, causing ibm17 to consistently exceed the 3600s harness cap and return a suboptimal early-exit placement. v2 tracks total elapsed time across all benchmarks and tightens remaining per-benchmark budgets accordingly.

ibm17 timeout eliminated · --all completes in ~480s
